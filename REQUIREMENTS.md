# Mithila Cultural Society Australia - Revised Website Scope

## Product Direction

Keep the website and platform simple, practical and easy to manage. The site should focus on the features that provide real community value instead of becoming a large multi-page application too early.

Use the organisation name exactly as **Mithila Cultural Society Australia**.

## Proposed Structure

### 1. Home Page

The home page should be the main public introduction page.

It can merge:
- Home
- About Us
- Contact summary
- Cultural introduction
- Committee/advisors preview
- Membership and donation prompts
- Upcoming event preview
- Gallery preview
- Blog preview

Contact details or a contact section should be visible or reachable from every page.

### 2. Events Page

The events page is the most important functional page.

It should support:
- View upcoming events
- View past events
- Event detail
- Event registration
- Pay for event tickets
- Admin-created events
- Event approval workflow
- Event status such as draft, pending approval, approved, published, completed and cancelled
- Admin as the final authority for approval and publishing
- Basic event management by authorised users

This page should provide real operational value for Mithila Cultural Society Australia.

### 3. Membership And Donations Page

Membership and donation should be combined into one simple page.

It should support:
- Annual membership
- Membership renewal
- Lifetime membership
- Donations
- Payment-ready structure
- Member application status
- Admin approval where required

### 4. Member Pages

Member pages are required but should remain simple.

Core member functions:
- Member dashboard
- My profile
- My membership
- My event registrations/tickets
- My donations/payments
- Announcements

### 5. Gallery Page

Gallery should be a single simple page.

It should support:
- Event photos
- Cultural images
- Album-style sections
- Admin-managed uploads in a later phase

### 6. Blog Page

Blog is required as one simple public page.

It should support:
- Society updates
- Cultural articles
- Event stories
- Admin-managed posts in a later phase

### 7. Contact

Contact does not need to be a large standalone section unless useful.

Contact should appear on every page through:
- Footer
- Contact block
- Email/phone/social links
- Optional contact form

## Simplified Navigation

Recommended top-level navigation:
- Home
- Events
- Membership & Donations
- Gallery
- Blog
- Member Portal
- Admin

Contact should be present globally rather than treated as a heavy separate feature.

## Admin Authority

Admin remains the final authority for:
- Approving events
- Publishing events
- Managing members
- Approving memberships where required
- Managing payments and donation records
- Managing gallery and blog content
- Publishing announcements

## Payment Scope

Payments should be planned but not fully integrated until the correct flow is approved.

Payment-ready areas:
- Annual membership
- Membership renewal
- Lifetime membership
- Donations
- Event tickets

Stripe can be used later for the actual payment implementation.

## Build Principle

Do not make the platform too elaborate. Prioritise:
- Clear one-page public presence
- Strong event management
- Simple membership/payment pathways
- Basic member portal
- Admin approval and governance
- Clean, mobile-first design
